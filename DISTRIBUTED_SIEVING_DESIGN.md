# YAFU Distributed Sieving — MVP Design

A server/client extension to YAFU's existing GGNFS sieving pipeline that lets
multiple machines contribute relations to a single factorization job. Modeled
loosely on CADO-NFS's workunit/SQLite design but built around `gnfs-lasieve4*`,
which YAFU already drives well.

## Goals

- Distribute lattice sieving across many trusted clients on a private network.
- Reuse the existing `nfs_sieving.c` Q-range bookkeeping and msieve-filtering
  pipeline; touch as little of YAFU's NFS code as possible.
- Be the kind of thing a small group of friends can spin up in an afternoon
  to attack a job too large for any single machine.
- Keep the server self-contained: one binary, one SQLite file, one directory
  of relation files. No daemons, no system DB, no TLS PKI to manage.

## Non-goals (MVP)

- TLS / PKI. HTTP only, shared bearer token. Document the trust model clearly.
- Untrusted clients. Verification exists for catching bugs, not adversaries.
- Windows clients. Linux only. (Windows could be added later with build work.)
- Multi-job scheduling. One factorization at a time per server.
- Auto-update of clients.
- Web UI. A `GET /stats` endpoint exists so a static HTML page can be added
  later, but the MVP ships no UI.
- Heartbeat / progress reporting. Lease timeout alone handles failure.
- NAT traversal. Clients must be able to reach the server's port directly.

## Architecture

```
   ┌─────────────────────┐
   │ yafu-sieve-server   │   one process, owns:
   │                     │     - job.db   (SQLite, metadata)
   │  HTTP API :8080     │     - rels/    (relation files)
   │  Verifier thread    │     - files/   (poly, optional .afb)
   │  Workunit dispatch  │
   └──────────┬──────────┘
              │ HTTP
   ┌──────────┴──────────┐──────────┬──────────┐
   │                     │          │          │
┌──┴──────────────┐  ┌───┴────┐ ┌───┴────┐ ┌───┴────┐
│yafu-sieve-client│  │ client │ │ client │ │ client │
│  shells out to  │  │        │ │        │ │        │
│ gnfs-lasieve4*  │  │        │ │        │ │        │
└─────────────────┘  └────────┘ └────────┘ └────────┘
```

The server replaces YAFU's role of dispatching Q-ranges to local threads. Each
client is a thin wrapper around an existing `gnfs-lasieve4*` binary that the
client operator has installed locally. The siever binary itself is **not**
shipped over the wire in the MVP; clients must have a matching version on disk.

The current `system("gnfs-lasieve…")` call site at `factor/nfs/nfs_sieving.c:2629`
is roughly what each *client* will do, just driven by remote workunits instead
of local thread-pool dispatch.

## Why this fits ggnfs cleanly

- ggnfs's `gnfs-lasieve4*` is fully self-contained: given a `.poly` (or `.job`)
  file and a Q-range, it generates its own factor base internally
  (`gnfs-lasieve4e.c:1438`) and writes a relation file. No precomputed `.afb`
  is required. We can ship just the `.poly` and the client is good to go.
  - Optional later optimization: pre-generate `.afb.0` once on the server
    (single `-F` invocation), ship it as a second file, save startup time
    per workunit.
- The relation file format is a plain text stream the existing msieve
  filtering already reads. Submitted relation files drop straight into the
  same directory and pipeline.
- YAFU already partitions sieving into Q-ranges. The server is essentially
  a remoted, durable version of the existing `nfs_sieve_dispatch` loop
  (`nfs_sieving.c:1068`). Relations come back as one file per workunit
  under `<jobdir>/rels/wu-*.dat`; the operator concatenates them into
  `nfs.dat` for downstream filtering (see `finalize-nfs.sh`).

## Protocol

### Auth

All endpoints require:

```
Authorization: Bearer <token>
```

The token is generated by the server operator (32 random bytes, hex) and
distributed manually to each client via its config file. Token mismatch
returns `401`.

No TLS in MVP. Token travels in plaintext over the LAN. **Document this
loudly in the README** — anyone on-path can replay the token, and submitted
relation files are also unencrypted.

### Endpoints

#### `POST /lease` — client requests work

Request:

```json
{ "client_id": "alice-box1", "client_version": "0.1.0" }
```

Response `200`:

```json
{
  "workunit_id": "wu-<jobhash>-000123",
  "q_start": 80000000,
  "q_range": 10000,
  "side": "a",
  "lease_seconds": 3600,
  "siever": "gnfs-lasieve4I14e",
  "command_template": "{siever} -f {q_start} -c {q_range} -a {job_file} -o {output_file} -n 0",
  "files": [
    { "name": "job.poly", "sha256": "abc123…", "url": "/file/abc123…" }
  ],
  "output": { "name": "rels.out", "max_bytes": 524288000 }
}
```

- `204 No Content` if no workunit currently available but the job is still
  running. Client backs off and retries.
- `410 Gone` if the job is complete. Client logs success and exits.

#### `GET /file/<sha256>` — fetch input files

Returns raw bytes; `Content-Length` set; `Content-Type: application/octet-stream`.
Client caches files locally by sha so a long session never redownloads
`job.poly`. `404` if sha is unknown to the server.

#### `POST /submit` — client returns relations

Headers:

```
X-Workunit-Id: wu-<jobhash>-000123
X-Client-Id: alice-box1
X-Sha256: <sha of body>
X-Sieve-Seconds: 1832.4
Content-Type: application/octet-stream
Content-Length: <bytes>
```

Body: raw relation file bytes (the contents of `rels.out`).

Response `200`:

```json
{ "accepted": true, "verified": "pending", "num_relations": 14823 }
```

- `409 Conflict` if the workunit was re-issued and another client's submission
  was already accepted. Client discards the result and re-leases.
- `400 Bad Request` on parse failure or sha mismatch. Client logs and re-leases.

#### `GET /stats` — for future stats page

Aggregate counters as JSON. The future stats page will be a static HTML file
that polls this endpoint.

```json
{
  "job_id": "<jobhash>",
  "rels_target": 250000000,
  "rels_total": 71200000,
  "rels_per_sec_1h": 12450.0,
  "workunits_total": 25000,
  "workunits_completed": 7120,
  "workunits_outstanding": 14,
  "workunits_failed": 3,
  "workunits_poisoned": 0,
  "clients": [
    { "id": "alice-box1", "last_seen": 1714142233,
      "total_relations": 12300000, "total_workunits": 1230 }
  ]
}
```

#### `GET /health` — liveness

```json
{ "ok": true, "job_id": "<jobhash>", "uptime_seconds": 38112 }
```

Client polls this on lease errors to distinguish "server restarted, my lease
is invalid" from "transient network failure."

### Workunit JSON

A workunit is the unit of work the server hands a client. Fields:

| field              | type    | purpose                                            |
|--------------------|---------|----------------------------------------------------|
| `workunit_id`      | string  | `wu-<jobhash>-<seq>`. Unique per server.           |
| `q_start`          | int     | special-q range start.                             |
| `q_range`          | int     | size of range.                                     |
| `side`             | "a"/"r" | algebraic or rational.                             |
| `lease_seconds`    | int     | server-configured per-WU expiry.                   |
| `siever`           | string  | required `gnfs-lasieve4*` binary name.             |
| `command_template` | string  | `{var}` substitutions client expands locally.      |
| `files`            | array   | input files (sha-addressed) the client must fetch. |
| `output.name`      | string  | local filename the siever writes to.               |
| `output.max_bytes` | int     | client aborts if siever output exceeds this.       |

The `command_template` design lets the server tweak siever flags (e.g.
`batch_3lp`, verbosity) without a client update.

### Workunit state machine

```
              lease
 available ─────────▶ leased
    ▲                  │
    │ lease_expires    │ submit
    │                  ▼
    └──────────────  submitted
                       │
            ┌──────────┴──────────┐
            │                     │
       verify_pass            verify_fail
            │                     │
            ▼                     ▼
        verified              failed
       (terminal)               │
                                │ attempt_count < max
                                ▼
                            available
                                │
                                │ attempt_count >= max
                                ▼
                            poisoned
                            (terminal,
                            operator alert)
```

All transitions are SQLite transactions. After 3 failed attempts (configurable)
a workunit goes to `poisoned` and surfaces in `/stats` so the operator notices.

## Storage

Two layers, kept strictly separate.

### SQLite — metadata only

Single file `job.db` in the server's working directory.

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;

CREATE TABLE workunits (
    id              TEXT PRIMARY KEY,
    q_start         INTEGER NOT NULL,
    q_range         INTEGER NOT NULL,
    side            TEXT NOT NULL CHECK (side IN ('a','r')),
    state           TEXT NOT NULL CHECK (state IN
                       ('available','leased','submitted','verified','failed','poisoned')),
    attempt_count   INTEGER NOT NULL DEFAULT 0,
    created_at      INTEGER NOT NULL,    -- unix seconds
    leased_at       INTEGER,
    leased_to       TEXT,                -- client_id
    lease_expires   INTEGER,
    completed_at    INTEGER,
    expected_rels   INTEGER
);
CREATE INDEX idx_wu_state ON workunits(state);
CREATE INDEX idx_wu_lease_expires
    ON workunits(lease_expires) WHERE state = 'leased';

CREATE TABLE submissions (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    workunit_id     TEXT NOT NULL REFERENCES workunits(id),
    client_id       TEXT NOT NULL,
    received_at     INTEGER NOT NULL,
    file_path       TEXT NOT NULL,       -- relative path under jobdir
    sha256          TEXT NOT NULL,
    num_relations   INTEGER NOT NULL,
    verify_status   TEXT NOT NULL CHECK (verify_status IN
                       ('pending','passed','failed','skipped')),
    verify_reason   TEXT,
    sieve_seconds   REAL                 -- client-reported, advisory
);
CREATE INDEX idx_sub_wu ON submissions(workunit_id);
CREATE INDEX idx_sub_pending
    ON submissions(verify_status) WHERE verify_status = 'pending';

CREATE TABLE clients (
    id                  TEXT PRIMARY KEY,
    first_seen          INTEGER NOT NULL,
    last_seen           INTEGER NOT NULL,
    total_relations     INTEGER NOT NULL DEFAULT 0,
    total_workunits     INTEGER NOT NULL DEFAULT 0,
    total_failures      INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE files (                     -- content-addressed input files
    sha256          TEXT PRIMARY KEY,
    path            TEXT NOT NULL,       -- on-disk path
    bytes           INTEGER NOT NULL,
    purpose         TEXT                 -- 'poly' | 'job' | 'afb'
);

CREATE TABLE meta (                      -- single-row job metadata
    key             TEXT PRIMARY KEY,
    value           TEXT
);
```

### Relation files on disk

Submitted relations land at `<jobdir>/rels/<workunit_id>.dat`. Easy to enumerate
for msieve filtering, easy to back up, and decoupled from the DB so the data
files survive even if the DB is corrupted.

### Crash recovery

On startup:

1. Open `job.db`.
2. `UPDATE workunits SET state = 'available' WHERE state = 'leased' AND lease_expires < now();`
3. Resume serving.

No replay log. SQLite's WAL handles in-flight transaction durability.

## Verification

A background verifier thread on the server polls
`SELECT … FROM submissions WHERE verify_status = 'pending'` and runs two
checks per submission:

1. **Parse pass (all relations).** Every line matches `a,b:rprimes:aprimes`,
   fields decode as hex, the embedded `q` is in `[q_start, q_start + q_range)`.
   Cheap, runs over the full file. Failure ⇒ `verify_status = 'failed'`.
2. **Spot-check K relations** (default `K=50`, configurable). Pick K random
   lines, recompute the rational norm `a − b·m` and the algebraic norm from
   the polynomial coefficients, trial-divide by the listed primes, confirm
   the remainder is `±1`. Any miss ⇒ `failed`.

We dropped "rate sanity" from earlier drafts: special-q yield naturally
varies a lot across a Q-range (prime-poor regions, large-prime fluctuations),
so a hand-tuned threshold either misses real bugs or false-positives on
honest variance. Parse + spot-check at K=50 is far stronger than rate for
both adversarial detection (P[miss] ≈ 1e-15 against a 50%-fake submitter)
and the realistic failure mode of a subtly broken siever build.

On `passed`: workunit ⇒ `verified`; `clients.total_relations += num_relations`.
On `failed`: workunit ⇒ `available` with `attempt_count += 1` (or `poisoned`
if at max); `clients.total_failures += 1`.

## Configuration

Server config (`yafu.ini`-style keys):

```
nfs_server_port=8080
nfs_server_token=<random 32-byte hex>
nfs_server_dbfile=job.db
nfs_server_jobdir=.
nfs_server_lease_seconds=3600
nfs_server_verify_spotcheck_k=50
nfs_server_max_attempts=3
nfs_server_local_clients=0          # auto-spawn N local yafu-sieve-client procs
nfs_server_default_qrange=10000
```

Client config:

```
nfs_client_server_url=http://host:8080
nfs_client_token=<same token>
nfs_client_id=<defaults to hostname>
nfs_client_siever_path=/usr/local/bin/gnfs-lasieve4I14e
nfs_client_workdir=/tmp/yafu-client
nfs_client_idle_backoff_seconds=30
```

## Libraries & build

Both binaries are pure C, in-tree dependencies, no system packages required.

| dep             | purpose                              | how shipped                       |
|-----------------|--------------------------------------|-----------------------------------|
| **mongoose**    | HTTP server (server) + client (cli)  | single `.c`/`.h`, MIT, vendored   |
| **cJSON**       | JSON encode/decode                   | single `.c`/`.h`, MIT, vendored   |
| **SQLite**      | metadata store (server only)         | amalgamation `.c`/`.h`, public domain, vendored |
| **GMP**         | norm computation for verification    | already a YAFU dependency         |

All three vendored libs are single-file drop-ins, which keeps the build story
simple and cross-platform-neutral. Add a `factor/nfs/distributed/` directory
holding `server.c`, `client.c`, `protocol.c`, `db.c`, `verify.c`, plus the
vendored libs, and wire two new Makefile targets: `yafu-sieve-server` and
`yafu-sieve-client`.

## Implementation plan

Phased so each step produces a testable artifact.

**Phase 0 — refactor the executor seam (in YAFU itself).**
Factor the `system("gnfs-lasieve…")` call at `nfs_sieving.c:2629` into a
small `executor` interface with one implementation: the existing local
`system()` path. Goal: zero behavior change, but the spot where a
"distributed executor" plugs in is now obvious. Verify by running an
existing local job to completion.

**Phase 1 — server walking skeleton.**
- `yafu-sieve-server` binary with `/lease`, `/file`, `/submit`, `/health`.
- SQLite schema, file-by-sha endpoint, no verification (`verify_status` =
  `skipped`), no stats endpoint yet.
- Workunits seeded from a one-shot CLI command:
  `yafu-sieve-server init --poly=foo.poly --qmin=… --qmax=… --qrange=…`

**Phase 2 — client walking skeleton.**
- `yafu-sieve-client` binary that polls `/lease`, fetches files (sha-cached),
  invokes `gnfs-lasieve4*` exactly the way YAFU does today, POSTs the result.
- Goal: end-to-end run of `init → server → 1 client → /submit → file on disk`.
  Manually inspect the relation file; pipe it to msieve filtering by hand
  to prove the format round-trips.

**Phase 3 — verification.**
- Verifier thread: parse pass on every relation, GMP spot-check on K random
  relations (default K=50). No rate check.
- `failed` ⇒ requeue with `attempt_count++` (or `poisoned` at max).

**Phase 4 — multi-client + lease expiry.**
- Multiple clients in parallel.
- Lease timeout requeues stuck workunits.
- `409` handling on duplicate submission.

**Phase 5 — quality of life.**
- `--local-clients=N` flag on the server: auto-spawns N local
  `yafu-sieve-client` processes connecting to `localhost`. Single-box use
  is now one command.
- `/stats` endpoint.
- README with token-generation instructions and the
  "this is HTTP, only run on a private network" warning.

**Phase 6 (post-MVP).**
- Static HTML stats page (`docs/stats.html` polls `/stats`).
- `.afb` pre-generation and shipping for faster client startup.
- Client capability advertising (which `gnfs-lasieve4I*` sievers it has)
  so a heterogeneous fleet can self-target.
- TLS / Windows / multi-job — only if real demand appears.

## Open questions / decisions remaining

1. **`workunit_id` format.** Current proposal: `wu-<jobhash>-<seq>`, where
   `<jobhash>` is the first 8 hex of `sha256(poly_contents)`. Lets two jobs
   coexist in the same DB without collision if we ever add multi-job.
2. **Client siever discovery.** If the client doesn't have the `siever`
   binary the server requested, it should `400` the lease (or refuse it
   client-side and return the lease via a future `/release` endpoint).
   For MVP I'd just have the client log and exit, operator's problem to
   install the right siever.
3. **`.poly` vs `.job` shipping.** YAFU currently passes a `.job` file with
   factor-base bounds embedded. Probably ship the `.job` (it's small) so
   the server controls all sieving parameters centrally. Confirm by
   tracing what `gnfs-lasieve4*` actually reads.
4. **Where does the server get the polynomial from?** Easiest: it's
   provided to `init`, copied into `<jobdir>/files/<sha>.poly`, and the
   `files` table points at it. The user already has it from polynomial
   selection.
5. **HTTP request size limit.** Submissions are typically MBs. mongoose
   handles streaming; pick a sensible cap (e.g. 1 GB) to keep memory
   bounded against a runaway client.

## Trust model summary

- Server and clients run on a network the operator controls.
- Single shared bearer token authenticates clients to the server.
- Plaintext HTTP. Anyone on-path can read submissions and replay tokens.
- Verification catches buggy/broken clients but is **not** an adversary
  defense — a hostile client that knows the math could submit lies that
  pass spot-check.
- All of the above is acceptable for the "small group of trusted operators"
  use case the MVP is targeting. Document loudly.
